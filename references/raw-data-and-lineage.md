# 原始数据表与溯源建模

## 目标

用户上传的原始 Excel/CSV 数据必须进入 Base，且后续过程表、结果表、异常表能回溯到原始数据。不要留下空白默认 `数据表`，也不要留下空白默认字段挤在原始字段前面。

## 原始数据进入 Base 的优先级

### 单个 Excel/CSV

优先用 Drive 导入：

```bash
lark-cli drive +import --file ./data.xlsx --type bitable --name "数据分析过程记录仓库-<主题>-<日期>"
```

导入完成后：

1. 从返回值取 Base token / URL。
2. 用 `lark-cli base +table-list --base-token <token>` 找到导入生成的原始表。
3. 把导入原始表作为事实表，不再重复创建一张 `04_核心明细` 来承载完全相同的原始数据。
4. 在原始表上创建视图：`全部原始数据`、`有效订单`、`退款订单`、`异常记录`、`分析样本` 等。
5. 在同一 Base 中追加分析过程表、结果表、洞察表和 dashboard。

### 多个 Excel/CSV 或复杂多工作表

如果无法一次导入到同一个 Base：

1. 先 `+base-create` 创建过程仓库。
2. 立即 `+table-list` 找到默认 `数据表`。
3. 用 `+table-update` 把默认 `数据表` 重命名为第一张原始表，例如 `原始数据_订单明细`。
4. 写入第一份原始数据前先处理默认字段。
5. 后续文件/工作表创建 `原始数据_<来源>` 表。
6. 每张原始表都要有 `来源文件`、`来源工作表`、`源行号` 或业务主键。

## 默认字段处理

飞书新建 Base/表时可能自动带出 `文本`、`单选`、`日期`、`附件` 等默认字段。原始数据表不能把这些字段空着放在前面，否则用户打开表会误以为原始数据没有写入。

处理顺序：

1. **复用默认主字段**：把默认 `文本` 字段改名为第一业务主键字段，例如 `订单 ID`、`客户 ID`、`源记录键`，并把该字段写入真实值。创建原始表时不要再重复创建同名主键字段。
2. **删除可删默认字段**：对确实无意义且接口允许删除的默认字段，用 `+field-delete` 删除。
3. **隐藏不可删字段**：若 OpenAPI 返回 `the method ... limited`，说明该字段删除或更新受限。不要反复重试；改用 `+view-set-visible-fields` 在原始数据视图里隐藏该字段。
4. **从源头覆盖字段结构**：能用 `+table-create --fields` 或 Drive 导入生成字段时，优先让字段结构一次成型，减少后续默认字段清理。

验收标准：

- `全部原始数据` 视图第一列应是有值的业务主键或 `源记录键`。
- 无业务意义且为空的默认字段不能出现在主要视图中。
- 如果因飞书接口限制无法删除字段，需要在过程说明中记录“已隐藏，接口限制不可删”。

## 视图设计

原始数据表通常至少创建这些视图：

| 视图名 | 目的 |
| --- | --- |
| `全部原始数据` | 保留完整事实表 |
| `有效记录` | 排除退款、取消、无效等记录 |
| `异常记录` | 缺失、重复、金额校验失败、极端值 |
| `分析样本` | 展示前 100/200 行，便于快速浏览 |
| `高风险记录` | 高退款、高金额、高延迟等业务风险 |

视图筛选、排序、分组分别用 `lark-base-view-set-filter.md`、`lark-base-view-set-sort.md`、`lark-base-view-set-group.md`。

## 溯源字段

所有原始表建议保留或新增：

| 字段 | 说明 |
| --- | --- |
| `源记录键` | 稳定唯一键；优先业务主键，否则 `文件名::工作表::行号` |
| `来源文件` | 本地文件名 |
| `来源工作表` | Excel sheet 或 CSV 文件名 |
| `源行号` | 原始行号 |
| `数据批次` | 一次分析任务的批次 ID |

## 关联关系

### 明细级结果

异常明细、清洗样本、重点订单清单等明细级表，建议创建 `link` 字段关联到原始表：

```json
{
  "name": "关联原始记录",
  "type": "link",
  "link_table": "原始数据_订单明细",
  "bidirectional": true,
  "bidirectional_link_field_name": "关联分析结果"
}
```

写记录时，关联值形如：

```json
{"关联原始记录": [{"id": "rec_xxx"}]}
```

### 聚合级结果

月度、SKU、地区、券型等聚合表不建议强行把每个聚合行关联到大量原始记录。更稳妥的溯源方式：

- 写明 `来源表`
- 写明 `聚合维度`
- 写明 `筛选条件`
- 写明 `来源行数`
- 写明对应原始数据视图或可复现条件

如果聚合行只对应少量原始记录，可以再加 `关联原始记录` link 字段。

## 记录 ID 映射

如果要建立 link 字段，需要保存原始表写入后的 `record_id_list`：

1. `+record-batch-create` 返回 `record_id_list`。
2. 按写入行顺序与 `源记录键` 建立映射。
3. 后续异常明细或结果表写入时，把对应原始记录 ID 写入 link 字段。

如果原始表来自 `drive +import`，需要用 `+record-list` 分页读取原始表，取回记录 ID 与业务主键/源记录键建立映射。

## Lookup 的使用边界

Lookup 适合从同 Base 的另一张表按条件取值或聚合，但不是建立关系的首选。需要可编辑关系时用 `link`；需要只读引用或聚合时，再读 `lookup-field-guide.md` 后创建 lookup 字段。
